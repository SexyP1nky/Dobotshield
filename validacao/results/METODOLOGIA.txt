
 BANCADA COMPARATIVA DE WAFs  --  METODOLOGIA
 DoBotShield  vs  ModSecurity (CRS)  vs  Coraza (CRS)

1. OBJETIVO

Avaliar, de forma reproduzivel e padronizada, a eficacia de tres Web
Application Firewalls protegendo duas aplicacoes propositalmente
vulneraveis, comparando-os a um baseline SEM WAF. Os WAFs avaliados:

  - DoBotShield  : reverse proxy WAF proprio (Go), com sanitizacao de
                   requisicao/resposta e rate-limiting/anti-bot.
  - ModSecurity  : nginx + OWASP Core Rule Set (CRS).
  - Coraza       : Caddy + OWASP Core Rule Set (CRS) (motor em Go).

Aplicacoes-alvo (open source, propositalmente vulneraveis):
  - DVWA   (PHP/MySQL)  -- vulnerables/web-dvwa
  - XVWA   (PHP/MySQL)  -- build local a partir de s4n7h0/xvwa
                           + mysql:5.7 (Xtreme Vulnerable Web Application)


2. MATRIZ DE TESTES
2 aplicacoes  x  4 cenarios de protecao  x  6 ferramentas  =  48 testes.

Cenarios por aplicacao:
  (a) no_waf       -- acesso HTTP direto a aplicacao (baseline)
  (b) modsecurity  -- atras de ModSecurity + CRS  (HTTPS)
  (c) dobotshield  -- atras do DoBotShield        (HTTPS)
  (d) coraza       -- atras de Coraza + CRS        (HTTPS)

Ferramentas (todas open source, executadas em containers):
  1. testssl.sh -- analise de criptografia TLS/SSL
  2. OWASP ZAP  -- DAST / vulnerability scanner (full-scan)
  3. SQLMap     -- SQL Injection
  4. XSStrike   -- XSS (fuzzing/evasao)
  5. Commix     -- Command Injection (OSCI)
  6. wrk        -- gerador de carga HTTP (teste de DoS/resiliencia)

4. PADRONIZACAO E JUSTICA EXPERIMENTAL
- Para CADA ferramenta, os parametros de ataque sao IDENTICOS nos quatro
  cenarios de uma mesma aplicacao. SO a URL de destino muda. Isso garante
  comparabilidade direta: nenhum cenario e "privilegiado".

- Vetores de ataque por ferramenta -- endpoint EXPLICITO em IP:porta; NENHUMA
  ferramenta usa --crawl, garantindo que o MESMO parametro vulneravel seja
  exercitado nos 4 cenarios de cada app (e que nada toque /setup/):
    * SQLMap  DVWA: GET  /vulnerabilities/sqli/?id=1&Submit=Submit  e
                         /vulnerabilities/sqli_blind/?id=1&Submit=Submit  (-p id)
    * SQLMap  XVWA: POST /xvwa/vulnerabilities/sqli/  --data "item=1"     (-p item)
    * XSStrike DVWA: GET /vulnerabilities/xss_r/?name=test
    * XSStrike XVWA: GET /xvwa/vulnerabilities/reflected_xss/?item=test
    * Commix   DVWA: POST /vulnerabilities/exec/  --data "ip=127.0.0.1&Submit=Submit"
    * Commix   XVWA: GET  /xvwa/vulnerabilities/cmdi/?target=127.0.0.1

- SQLMap (vetor calibrado p/ expor a falha do DoBotShield em SQLi):
    --technique=B (boolean-based blind) --tamper=equaltolike --string=<marcador>
    --level=1 --risk=1 --delay=0.3, IDENTICO nos 4 cenarios de cada app.
  Justificativas (todas medidas empiricamente contra o lab):
    * equaltolike troca '=' por LIKE -> evade a regra OR/AND-com-'=' do WAF e a
      injecao continua valida no MySQL (LIKE sem curinga == igualdade).
    * technique=B: a unica classe cujo payload (ex.: 1 AND 1 LIKE 1) NAO contem
      UNION/SLEEP/comentario/$() -> passa pelo DoBotShield. As demais (E,U,S,T)
      sao 100% bloqueadas (400) e so poluem.
    * --string=<marcador>: texto presente SO na resposta TRUE (DVWA=Surname,
      XVWA=Category). Faz o SQLMap decidir TRUE/FALSE pelo conteudo e IGNORAR o
      ruido do WAF (400 dos bloqueios + 502 da inspecao de resposta em erro SQL).
    * level 1 (e NAO o "nivel maximo"): ACHADO IMPORTANTE -> no level 5 o SQLMap
      dispara ~1400 payloads, a maioria vira 400/502, e esse mar de erros
      CORROMPE a calibracao TRUE/FALSE dele -> ele descarta ate o payload simples
      que passa. No level 1 manda poucos payloads, o que funciona nao e afogado e
      o --string crava. Confirmado: level 5/risk 3 e level 5/risk 1 NAO detectam;
      level 1/risk 1 detecta (DVWA 'id' e XVWA 'item' = vulnerable).
  Mesmo input nos 4 cenarios (so a URL muda) -> comparacao justa; mede a
  resistencia de cada WAF a uma injecao booleana ofuscada leve.

- Commix: --delay=1 em TODOS os cenarios (Commix pode derrubar a app sem
  throttle). Sem --crawl; alvo no endpoint vulneravel. A URL e passada pelo STDIN
  (echo <url> | docker run -i ... commix.py), NAO por --url. Motivo: rodando via
  "docker run" sem TTY, o Commix le alvos do stdin e ignoraria um --url (assim
  antes nao testava NADA). Passar a URL pelo stdin liga o STDIN_PARSING, que tem o
  efeito colateral DESEJADO de deixar o default do prompt "spawn pseudo-terminal
  shell?" como "n": ao confirmar a injecao (ex.: no_waf), o Commix NAO abre o shell
  interativo que, sem stdin real, entrava em loop e INUNDAVA o log (>1 GB). Tambem
  removido --all (enumeracao pesada, desnecessaria p/ medir deteccao). (bug duplo
  corrigido: "nao testava nada" + "flood de log de >1 GB").
  NOTA (resultado): o DoBotShield NAO falha no Commix. A brecha existe (echo/sleep
  nao estao na lista de comandos do WAF e executam), mas toda confirmacao do
  Commix usa $(...) (classic/eval/time) ou um interpretador da lista (python/perl)
  -> bloqueados. O file-based (unica via sem $()) exige diretorio web gravavel,
  que o DVWA nao tem. Logo o Commix e barrado pelo DoBotShield.

- XSStrike: 10 threads, --skip-dom, sem --crawl; alvo no parametro refletido.

- ZAP: full-scan autenticado no DVWA e com os mesmos parametros nos oito alvos.
  Para evitar que uma limitacao de memoria da bancada interrompa o processo Java,
  a regra ativa 40026 (DOM XSS) e marcada como IGNORE por zap_rules.tsv em TODOS
  os cenarios. A exclusao e uniforme e nao altera nenhum WAF; XSS refletido segue
  coberto pelo proprio ZAP e pelo XSStrike. Os alvos sao executados de forma
  sequencial, mantendo ativos apenas o backend e o WAF do cenario corrente.

- NOTA DE JUSTICA/EVASAO: o tamper (SQLMap) e os endpoints/payloads sao
  IDENTICOS no no_waf e nos 3 WAFs. Eles tornam o ataque mais realista (com
  ofuscacao leve) e sao aplicados igualmente -> a comparacao permanece justa;
  o que varia e apenas a capacidade de cada WAF de barrar o MESMO ataque.

- Os --delay (SQLMap 0.3s; Commix 1s) NAO sao alterados entre cenarios.

- wrk e, por definicao, um teste de carga/DoS: dispara sem delay (12 threads
  x 400 conexoes x 30s), em 3 repeticoes por cenario e identico em todos os
  cenarios. Mede throughput e
  resiliencia, nao deteccao de payload.

- DVWA exige autenticacao. Faz-se login automatico (admin/password), define-se
  DVWA Security = "low" e captura-se o cookie (PHPSESSID;security=low), injetado
  em SQLMap/XSStrike/Commix via --cookie/--headers. Para o ZAP o cookie e RENOVADO
  (login fresco) antes do scan e passado por VARIAVEL DE AMBIENTE (DVWA_COOKIE); o
  hook zap_hook.py cria a regra de Replacer pela API do ZAP
  (zap.replacer.add_rule, REQ_HEADER Cookie) injetando-o em TODA requisicao ->
  varredura AUTENTICADA das paginas internas. ATENCAO: criar a regra por
  "-config replacer.rules..." NAO funciona nesta versao do ZAP (config aceita mas
  nao aplicada) -> antes o scan ia DESLOGADO e so achava login.php (bug corrigido).
  O hook exclui logout/setup/security E TAMBEM /vulnerabilities/csrf (este ultimo
  TROCA a senha do admin durante o active-scan e quebraria o login das ferramentas
  seguintes). O reset/criacao do banco do DVWA ocorre somente no lab_00_setup. Os
  scanners renovam cookie com --no-setup, sem tocar em /setup.php.

- XVWA NAO exige login: os modulos de vulnerabilidade sao abertos, entao roda
  sem cookie. So e preciso criar o banco uma vez via GET /xvwa/setup/?action=do
  (feito no lab_00_setup). O ZAP exclui /xvwa/setup/ (hook) para nao resetar o
  banco durante a varredura.


5. CONFIGURACAO DOS WAFs

- ModSecurity / Coraza: OWASP CRS, paranoia 1, anomaly inbound 5 / outbound 4
  (CRS), modo bloqueio. TLS terminado pelo proprio WAF.

- DoBotShield: WAF em modo "block", inspecao de requisicao E resposta,
  e RATE-LIMITING LIGADO (RATE_LIMIT=10 req/s, BURST=20, MAX_CONNS=10 por IP).
  DECISAO METODOLOGICA: o rate-limiting/anti-bot e funcionalidade central do
  produto; portanto o DoBotShield e avaliado como em producao.

  Implicacao a registrar na analise:
   * No teste wrk (DoS), o rate-limiting estrangula o flood (HTTP 429) -- e a
     defesa relevante; ModSecurity/Coraza (CRS) NAO fazem rate-limiting.
   * SQLMap roda com --delay=0.3 (~3 req/s) e Commix com --delay=1 (~1 req/s),
     ambos abaixo do limite de 10 req/s. Eventuais bloqueios do DoBotShield no
     SQLMap/Commix podem ser por REGRA de WAF ou por taxa -- interpretar os logs
     (X-Shield-Action / eventos DoS_BLOCK vs WAF_BLOCK) ao analisar.
   * ZAP e XSStrike sao mais rapidos; contra o DoBotShield, parte dos
     bloqueios pode advir do rate-limiting alem das regras. Interpretar os
     logs (X-Shield-Action / eventos DoS_BLOCK vs WAF_BLOCK) ao analisar.

ANALISE DOS RESULTADOS (revisao 2026-06-06) -- 4 pontos verificados


PONTO 1 -- ZAP no XVWA gera poucos alertas (e a comparacao com WAF):
  Contagem de TIPOS de alerta (e instancias) extraida dos zap_report.json:
     XVWA  no_waf=23(42)  modsec=23(43)  dobot=19(37)  coraza=24(50)
     DVWA  no_waf=42(136) modsec=40(135) dobot=14(40)  coraza=38(134)
  Conclusao 1: dentro do XVWA, no_waf NAO gera "bem menos" que com WAF -- os
  numeros sao equivalentes (23 vs 23/24, dobot ate menor). A percepcao de
  "poucos alertas" e REAL mas o contraste verdadeiro e XVWA(~23) << DVWA(~42).
  Conclusao 2 (causa-raiz): o spider do ZAP quase nao percorre o XVWA. URLs
  unicas atacadas por cenario (instancias com param): XVWA so /xvwa/, /, robots,
  sitemap, .git/config (6 caminhos) -- NUNCA chega em /xvwa/vulnerabilities/sqli,
  /xss_*, /cmdi etc. Resultado: 0 HIGH reais no XVWA (o unico "High" =
  'Spring4Shell' em dobot/xvwa e FALSO-POSITIVO: XVWA e PHP, nao Java/Spring).
  No DVWA o ZAP entra autenticado (cookie via zap_hook.py) e varre 77 URLs ->
  6 HIGH no no_waf. "Total of N URLs" do spider: XVWA 11/6/11/6 vs DVWA 77/69/43/63.
  => O baixo numero no XVWA e COBERTURA do spider (nao bug de execucao). Para o
  ZAP exercitar de fato o XVWA, e preciso semear as URLs vulneraveis (seed/
  contexto) e/ou sessao -- isso ALTERA o input do ZAP so para o XVWA.

PONTO 2 -- DoBotShield ficou vulneravel a alguma ferramenta?
  SIM, apenas a UMA: SQLMap (SQLi booleano-cego). Verdicts dos logs:
     SQLMap  DVWA 'id'  -> "GET parameter 'id' is vulnerable"  (igual ao no_waf)
     SQLMap  XVWA 'item'-> "POST parameter 'item' is vulnerable"(igual ao no_waf)
     XSStrike-> "WAF detected"; gera 3072 payloads, NENHUM passa -> BLOQUEADO
     Commix  -> HTTP 400 (Bad Request) na fase de teste -> NAO injeta -> BLOQUEADO
     ZAP/testssl/wrk -> nao sao vetores de bypass (DAST/TLS/carga)
  ModSecurity e Coraza bloqueiam TODAS (incl. o SQLi). Logo, o unico "furo"
  do DoBotShield em toda a bancada continua sendo o SQLi booleano.

PONTO 3 -- Commix esta OK?
  SIM. 8/8 rodaram e percorreram o fluxo ate o veredito:
     no_waf  -> "parameter ... appears to be injectable (classic command
                injection)"; recusa pseudo-shell (n) e pula testes extras (Y)
                -> NAO ha mais o flood de >1GB (correcao do STDIN/--all mantida).
     modsec/coraza -> HTTP 403 (bloqueio CRS).
     dobotshield   -> HTTP 400 (bloqueio DoBotShield).
  Observacao (fluxo): nos 3 WAFs o Commix encerra cedo porque recebe 403/400 na
  fase de heuristica -- isso e o WAF bloqueando ANTES da injecao (resultado
  esperado), nao um defeito do Commix nem de input.

PONTO 4 -- Quais ferramentas estao 100% OK (sucesso absoluto, sem erro):
  100% LIMPAS (8/8, fluxo completo, coerente): SQLMap, XSStrike, Commix, wrk.
  OK COM RESSALVA ESPERADA:
     testssl -> 6/8 RC=0; 2/8 (no_waf, HTTP puro) RC=10 "not a TLS server".
                E POR DESIGN (baseline sem TLS), nao e erro.
     ZAP     -> 8/8 RC=0, mas a COBERTURA no XVWA e rasa (Ponto 1). Roda inteiro

  - SQLMap : responde "skip other DBMS? N" e "keep testing others? Y" -> segue. OK.
  - XSStrike: --skip evita o prompt e continua testando todos os payloads. OK.
